# Insurance Policy Ingestion Pipeline

Seven-step pipeline that converts insurance policy PDFs into enriched vectors stored in Qdrant, for use in RAG applications.

## What this solves

Insurance policy documents are dense, table-heavy PDFs that break naive chunking. Fixed-boundary splits discard table structure and strip context from individual chunks, which makes retrieval unreliable. This pipeline handles all three problems: tables are parsed and narrated by an LLM so they're retrievable by meaning, each chunk gets a context sentence anchoring it to its source document, and structured metadata is extracted and attached so Qdrant can filter by product type or insurer without touching the vector.

## Architecture

```
PDF
 │
 ▼
[Step 1] MinerU API ─────────────────────────── content_list.json + full.md
 │
 ▼
[Step 2] Chunker (200w / 50w overlap, bbox) ──── chunks.jsonl
 │
 ▼
[Step 3] Table Enrichment (Gemini LLM) ──────── chunks_with_tables.jsonl
 │
 ▼
[Step 4] Context Enrichment (Gemini LLM) ────── chunks_with_context.jsonl
 │
 ▼
[Step 5] One-Page Summary (Gemini LLM + PDF) ── one_page_summary.json
 │
 ▼
[Step 6] Metadata Extraction (Gemini LLM) ────── chunks_with_metadata.jsonl
 │
 ▼
[Step 7] Gemini Batch Embeddings (1536-dim) ──── Qdrant (product_summary)
```

## Pipeline steps

### Step 1: PDF extraction

MinerU converts each PDF into a structured content list (JSON) and a full-document Markdown file. Each element in the content list includes a bounding box — the pixel coordinates of that element on its PDF page. A deterministic `chunk_id` is assigned to each element here and carried through every subsequent step, so that any chunk surfaced at query time can be traced back to its exact location in the original document. This is what makes citation highlighting possible: a downstream UI can open the PDF and draw a highlight directly over the source passage.

### Step 2: Chunking

Content elements are merged into 200-word chunks with a 50-word overlap. Each chunk keeps its bounding-box coordinates from the source PDF, which makes page highlighting straightforward for any downstream UI. Tables are kept whole rather than split mid-row.

### Step 3: Table enrichment

MinerU outputs tables as raw HTML. This step sends each table to Gemini and replaces the HTML with two things: a hierarchical JSON representation of the table structure, and a short natural-language narrative. That narrative is what gets embedded, so table content is retrievable by meaning rather than by exact string.

**Before** (raw HTML from MinerU):

```html
<table>
  <tr><td>Vitality Status</td><td>Annual Premium Adjustment Percentage</td></tr>
  <tr><td>Bronze</td><td>+2%</td></tr>
  <tr><td>Silver</td><td>+1%</td></tr>
  <tr><td>Gold</td><td>-1%</td></tr>
  <tr><td>Platinum</td><td>-2%</td></tr>
</table>
```

**After** (HTML replaced by two fields):

```json
hierarchical_json = {
  "headers": {
    "column_headers": ["Annual Premium Adjustment Percentage"],
    "row_headers": ["Bronze", "Silver", "Gold", "Platinum"]
  },
  "rows": [
    {"row_header": "Bronze",   "cells": [{"column_header": "Annual Premium Adjustment Percentage", "value": "+2%"}]},
    {"row_header": "Silver",   "cells": [{"column_header": "Annual Premium Adjustment Percentage", "value": "+1%"}]},
    {"row_header": "Gold",     "cells": [{"column_header": "Annual Premium Adjustment Percentage", "value": "-1%"}]},
    {"row_header": "Platinum", "cells": [{"column_header": "Annual Premium Adjustment Percentage", "value": "-2%"}]}
  ]
}

table_narrative = {
  "purpose": "show_schedule",
  "dimensions": {"row": "Vitality Status", "col": "Annual Premium Adjustment Percentage"},
  "narrative": [
    "The Annual Premium Adjustment Percentage is determined by the Insured's Vitality Status.",
    "A Bronze Vitality Status results in a +2% adjustment to the annual premium.",
    "A Silver Vitality Status results in a +1% adjustment.",
    "A Gold Vitality Status results in a -1% adjustment.",
    "A Platinum Vitality Status results in a -2% adjustment."
  ]
}
```

The `table_narrative` is what gets embedded. A query like "what premium discount do Platinum members get?" matches "A Platinum Vitality Status results in a -2% adjustment" directly. The raw HTML produces a much weaker embedding with no semantic relationship between adjacent cells.

### Step 4: Context enrichment

This step implements Anthropic's [Contextual Retrieval](https://www.anthropic.com/news/contextual-retrieval) technique. Each chunk gets a short context prefix generated by sending the full document alongside the chunk to the LLM, using the prompt:

> *"Please give a short succinct context to situate this chunk within the overall document for the purposes of improving search retrieval of the chunk."*

Without this, a chunk that talks about "Activities of Daily Living" or "surrender charges after year 3" carries no signal about which product it belongs to or what part of the policy it covers. The context sentence anchors it.

**Examples from real pipeline output:**

| Chunk text (first 50 words) | Context added |
|---|---|
| *"lump sum, the death benefit under this basic policy. The TPD Benefit is an accelerated benefit that accelerates the death benefit..."* | Definition of Total and Permanent Disability (TPD) and its triggers based on age and Activities of Daily Living. |
| *"following 6 Activities of Daily Living even with the aid of special equipment, and always to require the physical assistance..."* | Definition of Total and Permanent Disability (TPD) after age 65 and specific immediate TPD conditions. |
| *"be considered a juvenile until such time he: (a) reaches the age of 16 and he is gainfully employed..."* | Definition of juvenile for TPD, underwriting limits, and term of the supplementary benefit. |
| *"to the policy anniversary occurring on or immediately following the Insured's 70th birthday. 2. Renewal Privilege..."* | Renewal and Conversion Privileges of the TPD on Secure Flexi Term (R&C) supplementary benefit. |

The context is prepended to the chunk text before embedding, so the vector captures both the specific content and its place in the document. Anthropic's research found this technique reduces retrieval failures by 49% compared to standard chunking. The full Markdown is cached server-side per document with a configurable TTL, so the many per-chunk LLM calls don't re-upload the same content each time.

### Step 5: One-page summary

The raw PDF is sent to Gemini alongside a prompt template to produce a structured product summary. Sending the PDF directly rather than extracted text avoids OCR artifacts in the summary and gives the model access to layout cues like headers and callout boxes.

### Step 6: Metadata extraction

Structured metadata (product type, insurer, coverage type) is extracted from the Step 5 summary and stamped onto every chunk. Storing it at the chunk level means Qdrant can filter by product type or insurer at query time without touching the vector.

### Step 7: Embedding and upsert

Chunks are embedded using `gemini-embedding-001` (1536 dimensions) via the Gemini Batch Embedding API. The text that gets embedded is `context + chunk text` combined — not the chunk text alone — so the resulting vector reflects both what the passage says and where it sits in the document. The step runs in two phases: all batch jobs are submitted first, then each job is polled and its vectors are upserted to Qdrant. Submitting before polling lets multiple documents be embedded concurrently rather than sequentially. A local registry tracks job state so completed batches are not resubmitted.

## Design decisions

- Idempotency at every step. Each step checks for existing output and skips completed work. The pipeline can be stopped and restarted without duplicating work. Step 4 resumes at the chunk level, so a partially-processed file picks up where it left off.
- Prompt caching in Step 4. The full document Markdown can be several hundred kilobytes. Caching it server-side avoids paying the upload cost on every per-chunk LLM call.
- Two-phase embedding. Submitting all jobs before polling any of them lets the Gemini Batch API process multiple documents concurrently rather than one at a time.
- Atomic file writes. All JSON and JSONL outputs are written atomically to avoid partial files that later steps could misread as complete.

## Skills demonstrated

RAG pipeline built from scratch, covering:

- **Chunking strategy**: overlap windows, bounding-box metadata preserved per chunk to support citation highlighting on the source PDF
- **Table-aware enrichment**: LLM-generated hierarchical JSON and natural-language narratives replace raw HTML tables, making structured data retrievable by meaning
- **Contextual Retrieval** (Anthropic technique): full-document context injected per chunk before embedding, reducing retrieval failures by 49%
- **Multi-stage LLM enrichment**: four distinct LLM calls per document covering table narration, chunk contextualisation, product summarisation, and structured metadata extraction
- **Gemini Batch Embedding API**: two-phase submit-then-poll design for concurrent batch job processing, with a local registry to avoid resubmission
- **Vector database design**: Qdrant collection setup, payload schema, upsert, and payload-filtered retrieval to narrow results by product type or insurer without touching the vector
- **Idempotent pipeline design**: each step resumes from existing output, with chunk-level resumption in Step 4, safe for long-running or interrupted workloads
- **Server-side prompt caching**: full document Markdown cached per document to avoid redundant uploads across many per-chunk LLM calls

## Setup

```bash
cd ingestion_pipeline
cp .env.example .env
uv sync
```

Set the following in `.env`:

| Variable | Used by |
|---|---|
| `MINERU_API_KEY` | Step 1 |
| `GEMINI_API_KEY` | Steps 3, 4, 5, 6, 7 |
| `QDRANT_URL` | Step 7 |
| `QDRANT_API_KEY` | Step 7 (optional) |

## Run

Process all PDFs in `../input`:

```bash
uv run python run_pipeline.py
```

Process a single PDF:

```bash
uv run python run_pipeline.py --file your_policy.pdf
```

Run steps individually:

```bash
uv run python step1_mineru_convert.py
uv run python step2_chunk_content.py
uv run python step3_table_enrich.py
uv run python step4_add_context.py
uv run python step5_one_page_summary.py
uv run python step6_add_document_metadata.py
uv run python step7_embed_and_upsert.py
```

Use `--force` on Steps 5, 6, and 7 to regenerate outputs even when cached results exist.
